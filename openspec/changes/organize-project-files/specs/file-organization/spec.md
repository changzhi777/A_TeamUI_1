# 项目文件整理规范

## 概述

整理归纳项目根目录下的非必要文件，将历史文档和临时脚本归档到对应文件夹，保持项目根目录整洁。

## 设计方案

### 目录结构

```
项目根目录/
├── .config/              # 集中的配置文件
├── archive/              # 归档目录
│   ├── docs/          # 历史文档
│   ├── reports/       # 历史报告
│   └── scripts/      # 历史脚本
├── docs/                 # 活跃项目文档（保留）
└── [其他标准项目文件]
```

## ADDED 需求

### 需求：历史文档归档结构

#### 场景：开发者需要查看历史项目文档

**给定** 项目根目录存在多个历史文档文件
**当** 开发者需要查找项目历史信息
**那么** 文档应归类在 archive/docs/ 目录下
**并且** 文档应按类型清晰组织
**原因** 当前散乱的文档使根目录不整洁，历史信息难以查找

### 需求：配置文件集中管理

#### 场景：需要修改项目配置

**给定** 配置文件分散在根目录
**当** 需要修改 eslint、knip、netlify 等配置
**那么** 所有配置文件应在 .config/ 目录下
**并且** 配置文件应易于查找和管理
**原因** 分散的配置难以统一管理和维护

### 需求：废弃目录清理

#### 场景：代码库包含已废弃的服务器代码

**给定** server/ 目录包含旧的服务端实现
**当** 项目已迁移到本地 MySQL，不再使用此代码
**那么** 应删除此目录以减少代码库体积
**并且** 删除后项目应正常构建和运行
**原因** 保留无用的废弃代码会增加仓库体积并引起混淆

### 需求：测试报告归档

#### 场景：需要查看历史测试结果

**给定** 测试报告散落在根目录
**当** 需要分析历史测试数据
**那么** 测试报告应归档在 archive/reports/ 目录下
**并且** 报告应按测试类型和时间组织
**原因** 历史测试数据未定期清理，占用根目录空间

### 需求：临时脚本归档

#### 场景：需要使用临时数据库脚本

**给定** create-db.js、test-db.js 等脚本在根目录
**当** 需要重新初始化数据库或测试
**那么** 脚本应归档在 archive/scripts/ 目录下
**并且** 脚本应有清晰的命名和说明
**原因** 根目录下的临时脚本是辅助工具，不应混杂在主目录

## 实施注意事项

1. **备份**：执行前确保 git 状态干净或已提交
2. **顺序**：先创建目录结构，再归档，后移动配置，最后删除
3. **验证**：每步后验证项目可正常构建和运行
4. **更新**：如需要，更新相关文档中的路径引用

## 验证标准

- 项目可正常构建：`pnpm run build`
- 项目可正常启动：`pnpm run dev`
- 根目录整洁：只保留必要文件和文件夹
